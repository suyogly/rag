## 18. Engineering Practices and Development Standards

Engineering at Sagarmatha Systems is structured around the principle that systems should remain understandable and maintainable over time, even as complexity increases. This principle influences how code is written, reviewed, deployed, and monitored.

All code must be version-controlled using the company’s designated repository system. Direct commits to the main branch are not permitted under any circumstances. All changes must go through a pull request process, which includes automated checks and manual review.

The expectation is not simply that code works, but that it can be understood by other engineers without requiring extensive explanation. Naming conventions, structure, and documentation within the codebase are considered part of the deliverable, not optional enhancements.

Code reviews are mandatory for all changes. A minimum of two approvals is required before a pull request can be merged. Reviewers are expected to evaluate not only correctness but also clarity, maintainability, and potential impact on existing systems. Reviews should be constructive and focused on improving the quality of the code rather than assigning fault.

Test coverage is required for all production code. While a minimum threshold of 80 percent coverage is enforced by automated tools, the quality of tests is considered more important than the percentage itself. Tests should validate meaningful behavior and edge cases rather than simply increasing coverage metrics.

Documentation must accompany significant changes, particularly when they affect system behavior or introduce new components. Documentation is maintained in the company’s central repository and should be updated as part of the development process.

## 19. Development Workflow

The standard development workflow follows a structured sequence to ensure consistency and traceability.

Work begins with a clearly defined task in the task management system. Tasks must include sufficient detail to allow implementation without requiring repeated clarification. If requirements are unclear, engineers are expected to seek clarification before beginning work.

Development occurs in feature branches, which are named according to a standardized convention that includes the task identifier and a brief description. This ensures traceability between code changes and task definitions.

Once development is complete, a pull request is created. The pull request must include a description of the changes, the reasoning behind them, and any relevant context for reviewers. Screenshots, logs, or test results may be included where appropriate.

Automated checks run on each pull request, including tests, linting, and security scans. Pull requests that fail these checks cannot be merged.

After review and approval, the changes are merged into the main branch. The merge triggers the deployment pipeline, which handles the progression of code through staging and production environments.

## 20. Continuous Integration and Deployment

Sagarmatha Systems uses a continuous integration and deployment pipeline to automate the build, test, and deployment processes. The pipeline is designed to ensure that code changes are validated and deployed in a controlled manner.

Each commit to a feature branch triggers a build process that includes compilation, testing, and static analysis. This process is intended to catch issues early in the development cycle.

When code is merged into the main branch, a more comprehensive pipeline is executed. This includes integration tests and deployment to the staging environment. The staging environment is intended to closely mirror production, allowing for realistic testing.

Deployment to production is controlled and typically occurs within designated deployment windows. These windows are established to minimize user impact and ensure that support staff are available in case issues arise.

In exceptional cases, deployments may occur outside of standard windows. Such deployments require approval from the engineering manager or a designated senior engineer.

Rollback procedures are defined for all services. Engineers must ensure that changes can be reversed in a controlled manner if issues are detected after deployment.

## 21. On-Call and Incident Response

To maintain system reliability, Sagarmatha Systems operates an on-call rotation for engineering teams. The purpose of the on-call system is to ensure that issues affecting production systems are addressed promptly.

Each team maintains its own on-call schedule, which is published in advance. Engineers are expected to be available during their on-call period and respond to alerts within defined timeframes.

Alerts are generated by monitoring systems when predefined thresholds are exceeded or anomalies are detected. These alerts are routed to the on-call engineer through designated communication channels.

Upon receiving an alert, the on-call engineer is responsible for assessing the situation, determining the severity, and initiating appropriate actions. This may include investigating logs, restarting services, or escalating the issue to other team members.

Incidents are categorized based on severity, which determines the expected response time and level of escalation. High-severity incidents require immediate attention and may involve multiple teams.

After an incident is resolved, a post-incident review is conducted. The purpose of this review is to identify the root cause, assess the response, and determine actions to prevent recurrence. Documentation of these reviews is maintained for future reference.

## 22. Quality Assurance and Testing

Quality assurance is integrated into the development process rather than treated as a separate phase. The QA team works closely with engineering teams to ensure that systems meet functional and non-functional requirements.

Testing includes multiple levels, including unit tests, integration tests, and end-to-end tests. Each level serves a specific purpose and contributes to overall system reliability.

Manual testing is conducted for scenarios that are difficult to automate or require human judgment. Automated testing is used for repetitive and predictable scenarios to ensure consistency and efficiency.

Test environments are maintained to replicate production conditions as closely as possible. This includes similar configurations, data structures, and dependencies.

Defects identified during testing are documented and tracked in the task management system. Engineers are responsible for addressing these defects in a timely manner.

## 23. Data Management and Privacy

Data management practices at Sagarmatha Systems are designed to ensure accuracy, security, and compliance with applicable regulations.

All data stored or processed by the company is classified based on sensitivity. Access to data is restricted according to classification, and employees are granted access only to the data necessary for their role.

Data should not be copied, shared, or transferred without authorization. Any transfer of data must follow established procedures to ensure security and compliance.

Backups are performed regularly to prevent data loss. Backup procedures are tested periodically to ensure that data can be restored if needed.

Data retention policies define how long data is stored and when it should be deleted. These policies are based on legal requirements and business needs.

Employees must report any suspected data breaches or unauthorized access immediately. Prompt reporting is essential to minimize impact and comply with regulatory obligations.

## 24. Hiring and Recruitment Process

The hiring process at Sagarmatha Systems is structured to identify candidates who possess both technical competence and alignment with the company’s working style.

The process begins with a review of applications, focusing on relevant experience and demonstrated skills. Selected candidates are invited to participate in a technical assessment, which may include coding exercises or system design tasks.

Candidates who perform well in the technical assessment proceed to one or more interviews. These interviews evaluate problem-solving ability, communication skills, and cultural fit.

Final decisions are made by a panel that includes representatives from engineering and operations. The goal is to ensure a balanced evaluation and reduce individual bias.

Offer letters include detailed information about compensation, role expectations, and employment terms. Candidates are given a reasonable period to review and accept the offer.

## 25. Onboarding Process

New employees undergo a structured onboarding process designed to help them integrate into the company effectively.

The onboarding process includes an introduction to company policies, tools, and systems. New hires are provided with access to necessary resources and assigned a mentor to guide them during the initial period.

The first few weeks focus on understanding the codebase, development workflow, and team dynamics. New employees are gradually assigned tasks to build familiarity and confidence.

Regular check-ins are conducted during the onboarding period to address questions and provide feedback. The goal is to ensure that new hires are able to contribute effectively within a reasonable timeframe.

## 26. Probation Period

All new employees are subject to a probation period, typically lasting six months. During this time, performance and fit are evaluated to determine whether the employment relationship should continue.

Probation includes formal reviews at the three-month and six-month marks. These reviews assess technical performance, communication, and alignment with company values.

Employees who meet expectations are confirmed in their role. Those who do not may have their probation extended or their employment terminated, depending on the circumstances.

The probation period is also an opportunity for employees to assess whether the company meets their expectations.

## 27. Career Development and Promotions

Sagarmatha Systems supports career development through a combination of formal and informal mechanisms. Employees are encouraged to take ownership of their growth and seek opportunities to expand their skills.

Promotions are based on demonstrated performance, increased responsibility, and consistent contribution to team objectives. There is no fixed timeline for promotion; decisions are made based on readiness rather than tenure.

Managers are responsible for discussing career goals with their team members and providing guidance on how to achieve them. Training and learning resources are made available to support development.

Employees may also take on additional responsibilities or move between teams, subject to organizational needs and approval.

## 28. Internal Communication Practices

Effective communication is essential for coordination and decision-making. Sagarmatha Systems emphasizes clarity and documentation in communication.

Employees are expected to communicate updates, issues, and decisions in a timely manner. Important information should be documented in a persistent format to ensure accessibility.

Meetings should have a clear purpose and agenda. Outcomes and decisions should be recorded and shared with relevant stakeholders.

Digital communication tools should be used appropriately, with consideration for clarity and professionalism. Messages should be concise and focused on relevant information.

## 29. Customer Interaction and Support

Employees who interact with customers are expected to maintain a professional and responsive approach. Customer communication should be clear, accurate, and timely.

Support requests are tracked through a dedicated system, which ensures that issues are documented and addressed systematically. Response times are defined based on the severity and nature of the request.

Engineering teams may be involved in resolving complex issues that require technical expertise. Coordination between support and engineering is essential to ensure effective resolution.

Customer feedback is collected and analyzed to identify areas for improvement. This feedback is used to inform product and process decisions.

## 30. Documentation Standards

Documentation is considered a critical component of all work at Sagarmatha Systems. It ensures continuity, reduces dependency on individuals, and supports onboarding.

All systems, processes, and significant decisions must be documented. Documentation should be clear, accurate, and kept up to date.

The company maintains a centralized documentation platform where all materials are stored. Employees are responsible for contributing to and maintaining documentation relevant to their work.

Documentation should be written with the assumption that the reader has no prior context. This approach improves accessibility and usability.
